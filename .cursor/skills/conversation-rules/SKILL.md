Cursor Rules — Steam Crawler Project
1. 总体原则（必须遵守）
本项目为长期运行的数据采集工程，不是一次性脚本
稳定性、可维护性优先于速度
所有实现必须模块化、可测试、可替换
禁止在任何阶段引入：
浏览器自动化
登录态模拟
反爬绕过行为
2. 代码结构与模块化要求
2.1 强制模块拆分
代码必须拆分为职责单一的模块，不得出现“全写在一个文件”的实现。

每个模块必须满足：

只做一类事情
对外通过清晰接口交互
内部实现可独立修改而不影响其他模块
典型模块包括（名称可调整，职责不可合并）：

启动 / 入口模块
配置管理模块
目录（AppID）获取模块
任务调度 / 状态管理模块
详情采集模块
限速与重试模块
数据存储模块
日志与监控模块
2.2 模块边界规则
模块之间 禁止直接访问内部状态
只能通过：
明确的方法
明确的数据结构
不允许：
隐式依赖
全局共享可变状态
3. 编码规范（抽象级）
函数只做一件事
单个函数应：
易于单元测试
无副作用或副作用可控
所有异常必须：
被捕获
被记录
不影响主流程持续运行
4. 配置与环境规则
所有可变参数必须集中管理（如限速、重试次数、存储路径）
禁止：
硬编码环境相关参数
在代码中直接写死运行策略
支持：
开发 / 生产运行模式切换
5. 长期运行与稳定性要求
程序必须支持：
断点续跑
异常后恢复
单模块失败不导致整体崩溃
所有网络请求必须：
有超时
有失败处理策略
有速率控制
6. 日志与可观测性规范
必须区分以下日志类型：
运行流程日志
错误 / 异常日志
数据统计日志
日志应：
可按时间追踪
不依赖控制台唯一输出
不得吞掉异常（即 catch 后无记录）
7. 文档（Docs）强制同步规则 ✅【重点】
7.1 文档更新是强制行为
每一次代码修改，都必须同步修改文档。

不允许：

只改代码不改文档
用“代码即文档”作为理由跳过
7.2 Docs 结构要求
Docs 至少包含以下文件（可扩展）：

docs/architecture.md

系统整体结构
模块关系说明
docs/modules.md

各模块职责
输入 / 输出说明
docs/changes.md ✅【必须维护】

按时间记录所有修改
7.3 docs/changes.md 编写规范（强制）
每次修改代码后，必须在 docs/changes.md 追加一条记录，包含：

修改日期
修改模块
修改动机（为什么改）
修改影响（影响了哪些行为或接口）
是否需要迁移 / 注意事项（如有）
示例结构（仅结构，不填具体内容）：

markdown

## YYYY-MM-DD
- Modules:
  - xxx
- Reason:
  - …
- Summary:
  - …
- Impact:
  - …
- Notes:
  - …
8. 修改行为约束（Cursor 行为规则）
在 Cursor 中执行任何修改时：

先说明将要修改哪些模块
再进行代码修改
最后更新 docs，并明确指出更新了哪些文档文件
如果无法同步更新文档，必须：

明确说明原因
提出后续补文档计划
9. 禁止行为（硬性约束）
禁止一次性大改无说明
禁止跨模块“顺手修改”
禁止未记录的行为变更
禁止为了“跑得快”牺牲稳定性
10. 最终目标共识
本项目目标是：

一个可被他人接手、可长期运行、可审计、可演进的工程系统

任何实现决策都必须服务于这一目标。
