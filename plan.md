Steam 平台爬取长期运行方案（无代码版执行计划）
一、总体目标
构建一个可长期无人值守运行的 Steam 数据采集系统，用于：

获取 Steam 全量 App 目录
识别真实“游戏”（非 DLC / 工具）
定期拉取并更新游戏详情数据
数据结构稳定、可恢复、可扩展
不依赖账号登录，不做反爬绕过
二、整体架构（逻辑层面）
系统拆分为 6 个独立模块，每个模块都可以单独测试、重启、替换。


启动器
 ├── 目录获取模块（AppID List）
 ├── 任务调度模块
 ├── 详情采集模块
 ├── 限速与失败控制模块
 ├── 数据存储模块
 └── 监控与日志模块
三、模块详细规划（重点）
1️⃣ 目录获取模块（只做一件事）
职责

从 Steam 官方接口获取完整 AppID 列表
保存为本地“目录快照”
设计要求

可手动触发
结果可序列化存储（JSON / DB）
能对比上一次结果，识别：
新增 AppID
已移除 AppID（可选）
成功标准

能稳定拿到 >10 万条 AppID
不依赖任何 cookie 或登录
2️⃣ 任务调度模块（核心）
职责

控制“什么时候、处理哪个 AppID”
防止一次性请求过多
设计思路

维护一个任务队列，状态至少包括：
pending（待处理）
success（成功）
failed（失败）
skipped（不符合条件）
建议能力

支持断点续跑
支持每日 / 每周重新刷新旧数据
支持失败重试（次数有限）
3️⃣ 详情采集模块（数据源）
职责

根据 AppID 请求 Steam 商店详情接口
只做“获取 + 解析”，不管存储
必须解析的核心字段

success 状态
type（game / dlc / application）
是否免费
发售日期
支持语言
价格（若存在）
标签 / 分类（若存在）
过滤逻辑

非 game 类型直接标记为 skipped
success=false 记录但不反复请求
4️⃣ 限速与失败控制模块（稳定性关键）
目标

确保脚本可以 连续跑几天甚至几周
要求

全局请求速率限制（例如 X 秒 / 次）
针对 HTTP 状态码区分处理策略：
200：正常
429：延迟 + 重试
5xx：延迟 + 限次重试
异常必须被捕获，不能中断主进程
额外建议

随机微小延迟（非必须，但更稳）
5️⃣ 数据存储模块（可维护性优先）
职责

将“清洗后的结构化数据”写入存储
建议设计

原始响应（raw）与结构化数据分开存
支持以下操作：
根据 AppID 更新
标记更新时间
快速查询是否已处理
存储形式（任选）

SQLite（单机长期跑非常合适）
JSON + 索引文件（简单但体量会大）
后期可迁移数据库，不影响采集逻辑
6️⃣ 监控与日志模块（长期运行必备）
日志至少分三类

运行日志（启动、停止、循环）
错误日志（异常、失败请求）
统计日志（成功数 / 失败数 / 跳过数）
建议输出

控制台
本地日志文件（按日期滚动）
额外（可选）

定期输出当前进度
简单健康状态提示（例如每小时）
四、运行策略（非常重要）
推荐运行模式
第一次运行

只跑一次全量 AppID
速度慢一点，稳定优先
后续长期运行

每天检查新增 AppID
每 N 天刷新一次旧游戏数据
永远不全量重爬
五、部署与运行环境建议
运行机器

不需要高性能
稳定网络 > 带宽
允许长期运行
系统层面

支持：
自动重启脚本
异常退出后恢复
不依赖 GUI
六、明确“不要做”的事情（给 AI 限边界）
不使用浏览器自动化（Playwright / Selenium）
不模拟登录
不破解 JS
不并发高频请求
不绕过反爬机制
七、阶段性交付与验收标准
阶段 1（基础可跑）
✅ 能获取 AppID
✅ 能获取单个游戏详情
✅ 不崩溃

阶段 2（可长期跑）
✅ 断点续跑
✅ 日志清晰
✅ 限速稳定

阶段 3（数据可靠）
✅ 数据结构稳定
✅ 可重复更新
✅ 错误可追溯
